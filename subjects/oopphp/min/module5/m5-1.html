Introductie inheritance

Het Probleem

Naarmate je applicatie groeit, krijg je vaak te maken met klassen die "alles" proberen te doen. In ons Blackjack spel hebben we een Player en een Dealer. Ze lijken op elkaar (beiden hebben kaarten), maar gedragen zich anders.

Een veelgemaakte fout is om al deze logica in één gigantische class te proppen.

Voorbeeld: De "God Class" Player

Kijk eens naar deze code. We proberen zowel de menselijke speler als de dealer in één Player class te vangen:

class Player
{
    private string $role; // 'human' of 'dealer'
    private array $hand = [];
    private int $balance = 1000; // Alleen voor humans

    public function __construct(string $role)
    {
        $this->role = $role;
    }

    public function playTurn(): void
    {
        if ($this->role === 'human') {
            // Wacht op input van gebruiker (Hit of Stand)
            // Mag splitsen of verdubbelen
        } elseif ($this->role === 'dealer') {
            // Automatische regels:
            // Moet kaarten pakken tot score 17 is
            // Mag NIET splitsen of verdubbelen
        }
    }

    public function placeBet(int $amount): void
    {
        if ($this->role === 'dealer') {
            throw new Exception("De dealer zet geen geld in!");
        }
        $this->balance -= $amount;
    }
}

Waarom is dit slecht?

  • Onnodige checks: Je code staat vol met if ($role === 'dealer') statements.
  • Overbodige properties: De Dealer heeft geen $balance nodig, maar sleept die property wel overal mee naartoe.
  • Kwetsbaar: Als je iets wijzigt voor de Dealer, breek je per ongeluk misschien iets voor de Speler.
  • Overtreding SRP: De class heeft niet één verantwoordelijkheid, maar probeert iedereen te zijn.

Dit noemen we een God Class (of "Blob"): een class die te veel weet en te veel doet. Het is een bekend "Anti-Pattern" in programmeren.

De Oplossing: Inheritance (Overerving)

In plaats van één grote class met if-statements, kunnen we specifieke classes maken voor elke rol. Maar we willen óók geen code dupliceren (zoals het beheren van de kaarten).

Dit is waar Inheritance (Overerving) van pas komt. We maken een basis class voor de gedeelde eigenschappen, en specifieke classes voor het unieke gedrag.

// De structuur waar we naartoe werken:

class Player { ... }         // Basis (kaarten, score)
class Human extends Player { ... } // Specifiek (geld, input)
class Dealer extends Player { ... } // Specifiek (regels)
