Interne Logica: Protected Methods

Waarom Protected Methoden?

Soms heb je ingewikkelde logica die door zowel de Parent als de Child classes gebruikt moet worden, maar die niet bedoeld is voor de buitenwereld (zoals index.php).

  • Private: Child classes kunnen er niet bij (te strikt).
  • Public: Iedereen kan het aanroepen (te los, onveilig).
  • Protected: Precies goed: alleen familie mag het gebruiken.

Voorbeeld: Score Berekening

Stel dat het berekenen van de score in Blackjack ingewikkeld is (rekening houden met Azen die 1 of 11 kunnen zijn). Zowel de Player als de Dealer moeten dit doen.

We maken een protected hulpmethode in de Parent class:

class Player
{
    protected array $hand = [];

    // Protected: Alleen voor intern gebruik door Player en Dealer
    protected function calculateHandValue(): int
    {
        $value = 0;
        $aces = 0;
        foreach ($this->hand as $card) {
            // ... complexe logica ...
        }
        return $value;
    }

    // Public: Dit is wat de buitenwereld gebruikt
    public function getScore(): int
    {
        // We roepen intern de protected methode aan
        return $this->calculateHandValue();
    }
}

Hergebruik in Child Classes

Nu kan de Dealer deze complexe logica hergebruiken, of zelfs uitbreiden, zonder dat hij de code opnieuw hoeft te schrijven.

class Dealer extends Player
{
    public function decideMove(): string
    {
        // De Dealer mag de protected methode gewoon gebruiken!
        // Hij gebruikt dit om te beslissen of hij nog een kaart moet pakken.
        $currentScore = $this->calculateHandValue();

        if ($currentScore < 17) {
            return "hit";
        }
        return "stand";
    }
}

Veiligheid voorop

Als je in index.php probeert om $dealer->calculateHandValue() direct aan te roepen, krijg je een Fatal Error. En dat is goed! Je dwingt de programmeur om de nette publieke methodes (zoals getScore) te gebruiken, terwijl de Dealer intern wel bij de rekenmachine kan.
