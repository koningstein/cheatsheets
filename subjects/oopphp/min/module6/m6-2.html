Abstracte Methoden

Het Contract

Een abstracte class dwingt structuur af. Soms weet de Parent class dat iets moet gebeuren, maar nog niet hoe.

In Blackjack moet elke speler een keuze maken (Hit of Stand). Maar de logica verschilt:

  • Dealer: Volgt vaste regels (moet hitten onder de 17).
  • Human: Wacht op input van de gebruiker.

Abstracte Methode Definiëren

In de abstracte Parent class definiëren we de methode zonder body (geen accolades {}). Dit noemen we een abstracte methode. Het fungeert als een contract: elke Child class die hiervan erft, MOET deze methode implementeren.

abstract class Player
{
    // Het contract: "Iedereen moet een beslissing kunnen nemen"
    // Let op de puntkomma aan het eind, geen body!
    abstract public function makeDecision(): string;

    // Normale methodes mogen ook nog steeds
    public function addCard(Card $card) { ... }
}

Implementatie in Child Classes

Als je nu de Dealer class maakt en erft van Player, zal PHP een foutmelding geven totdat je de methode makeDecision daadwerkelijk schrijft.

class Dealer extends Player
{
    // Verplichte implementatie
    public function makeDecision(): string
    {
        // De Dealer logica
        if ($this->getScore() < 17) {
            return 'hit';
        }
        return 'stand';
    }
}

En voor de menselijke speler ziet de implementatie er heel anders uit:

class HumanPlayer extends Player
{
    // Verplichte implementatie
    public function makeDecision(): string
    {
        // De Human logica (bijv. uit formulier input)
        return $_POST['action'] ?? 'stand';
    }
}

Polymorfisme in Actie

Dankzij abstracte methoden weet je zeker dat elk Player-object een makeDecision() methode heeft. De Game class hoeft niet te weten welk type speler het is:

foreach ($this->players as $player) {
    // Polymorfisme: We roepen dezelfde methode aan,
    // maar er wordt andere code uitgevoerd afhankelijk van het object.
    $action = $player->makeDecision();

    if ($action === 'hit') { ... }
}

Regels voor Abstracte Methoden

  • Kunnen alleen bestaan in een abstracte class.
  • Mogen geen body hebben (eindigen met ;).
  • Child classes moeten ze implementeren met dezelfde signature (naam, parameters en return type).
  • Visibility (public/protected) mag in de child class niet strenger worden (wel soepeler).
