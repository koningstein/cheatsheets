Private Properties: De Kluis

Tot nu toe hebben we bijna overal public voor gezet. Dat is makkelijk, want dan kun je overal bij. Maar in het echte leven is niet alles openbaar. Je pincode, je dagboek of de inhoud van je portemonnee zijn privé.

In programmeren is dat net zo. Soms wil je data beschermen zodat niemand er per ongeluk (of expres) mee kan rommelen. Daarvoor gebruiken we private.

Public vs. Private

Dit noemen we de Zichtbaarheid (Visibility) van je code.

  • Public (Openbaar): Iedereen mag de waarde zien én veranderen. De voordeur staat wagenwijd open.
  • Private (Privé): Alleen de class zélf mag bij de data. De deur zit op slot en alleen de class heeft de sleutel.

Het Probleem: Valsspelen in Blackjack

Stel je voor dat we de score van een speler bijhouden. Als we dit public maken, kan iedereen de score aanpassen. Een programmeur kan (per ongeluk) een fout maken, of een hacker kan valsspelen.

class Player
{
    public int $score = 0;
}

$player = new Player();
// Omdat het public is, mag dit gewoon:
$player->score = 1000; // Valsspelen!
$player->score = -50;  // Fout! Een score kan niet negatief zijn.

Dit willen we voorkomen. We willen controle hebben over wat er met de score gebeurt.

De Oplossing: Encapsulatie (Inkapseling)

We gaan de data beschermen door de property private te maken. Dit noemen we met een duur woord Encapsulatie.

De Vergelijking: De Auto

Denk aan een auto. De motor is het belangrijkste onderdeel.

  • Je kunt de motor niet zomaar aanraken tijdens het rijden (hij is private / afgeschermd).
  • Je gebruikt pedalen en een stuur om de auto te besturen (de public methodes).

Als je direct aan de motor zou kunnen sleutelen tijdens het rijden, zou de auto stuk gaan. Door de motorkap dicht te laten (private), blijft de auto veilig werken.

Hoe gebruik je Private Properties?

Als je een property private maakt, kan niemand er meer bij. Als je het probeert, krijg je een error.

class Player
{
    private int $score = 0; // Nu met 'private'
}

$player = new Player();
// echo $player->score;
// FOUTMELDING: Fatal Error: Cannot access private property

Maar wacht... als niemand erbij kan, hebben we er toch niks aan?
Klopt! Daarom maken we speciale "deurtjes" om op een veilige manier naar binnen te gaan: Getters en Setters.

De Poortwachters: Getters en Setters

Omdat de property op slot zit, maken we twee methodes die wél public zijn. Zij hebben de sleutel van de kluis.

1. De Setter (De Bewaker)

Met de setter kunnen we de waarde veranderen. Het grote voordeel is dat we hier regels (validatie) kunnen toevoegen.

public function setScore(int $newScore): void
{
    // REGEL 1: Score mag niet negatief zijn
    if ($newScore < 0) {
        echo "Fout: Score mag niet negatief zijn!";
        return; // Stop de functie
    }

    // REGEL 2: Score mag niet hoger zijn dan 21 (in dit voorbeeld)
    if ($newScore > 21) {
        $newScore = 21;
    }

    // Als alles klopt, passen we de private variabele aan
    $this->score = $newScore;
}

2. De Getter (De Etalage)

Met de getter kunnen we de waarde bekijken. We mogen alleen kijken, niet aanraken.

public function getScore(): int
{
    return $this->score;
}

Het complete plaatje

Hier zie je hoe het veilig werkt:

$player = new Player();

// 1. Proberen vals te spelen (Foute waarde)
$player->setScore(-100);
// Output: Fout: Score mag niet negatief zijn!
// (De score is nog steeds 0)

// 2. Geldige score instellen
$player->setScore(10);

// 3. Score bekijken
echo "De score is: " . $player->getScore();
// Output: De score is: 10

Waarom doen we dit?

Misschien lijkt het veel extra werk om voor elke property getters en setters te maken. Maar dit is dé manier om professionele code te schrijven.

  • Je beschermt je data tegen fouten.
  • Je kunt later makkelijk regels toevoegen in de setter, zonder dat je de rest van je code hoeft aan te passen.
