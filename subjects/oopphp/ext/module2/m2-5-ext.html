Samenvatting: Professioneel Werken

Gefeliciteerd! Je hebt Module 2 afgerond. Waar Module 1 ging over "hoe schrijf ik code", ging deze module over "hoe organiseer ik mijn project".

Je bent overgestapt van de hobby-manier (alles los in één map) naar de professionele manier (mappenstructuur, namespaces en tools). Laten we de balans opmaken.

1. De Evolutie van het Laden

We hebben in deze module een flinke stap gemaakt in hoe we bestanden koppelen.

Niveau De Methode Waarom was dit (niet) handig?
Amateur require 'Card.php';
require 'Deck.php';		 ❌ Nadeel: Je moet elk bestand handmatig typen. Bij 50 bestanden word je gek. Verander je een naam? Dan crasht alles.
Gevorderd spl_autoload_register() ✅ Voordeel: PHP zoekt zelf de bestanden erbij.
❌ Nadeel: Je moet zelf een ingewikkelde functie schrijven die rekening houdt met mapjes.
Prof (Composer) require 'vendor/autoload.php'; ✅ Voordeel: De standaard in de wereld. Composer regelt alles. Het is snel, foutloos en werkt met namespaces.

2. Namespaces: Orde scheppen

Omdat we professioneel werken, gebruiken we Namespaces. Dit zijn de "achternamen" van je classes.

De Kern

  • Probleem: Jij hebt een class Card. Een gedownloade betaal-plugin heeft óók een class Card. PHP raakt in de war.
  • Oplossing: Je geeft jouw class een achternaam: Blackjack\Card. De andere heet Payment\Card.
  • Gevolg: Geen ruzie meer tussen bestanden.

3. Composer: De Projectmanager

De grootste ster van deze module is Composer. Dit is niet zomaar een tooltje, het is de manager van je project.

  • composer.json: Het contract. Hierin staat: "Mijn project heet Blackjack, mijn code staat in de map src/".
  • vendor map: Het magazijn. Hier stopt Composer alle spullen die hij nodig heeft (zoals de autoloader). Hier kom je zelf nooit aan.
  • autoload.php: De sleutel. Door alleen dit ene bestand in te laden, heb je toegang tot alles.

4. De Nieuwe Mappenstructuur

Je project ziet er nu heel anders uit dan in het begin. Alles heeft een vaste plek.

Mijn-Blackjack-Project/
├── src/               <-- HIER woon jij. Al jouw classes (Card, Player) staan hier.
├── vendor/            <-- HIER woont Composer. Blijf hier weg!
├── index.php          <-- De voordeur. Hier start het spel.
├── composer.json      <-- Het boodschappenlijstje.
└── .gitignore         <-- De regel: "Zet de vendor-map niet op GitHub".

5. Waarom Bulk Setters?

Een opvallende wijziging in deze module was dat we de Constructor hebben ingeruild voor Bulk Setters (zoals setCard()).

Waarom deden we dat?

  • Een constructor (new Card(...)) kan niks "terugzeggen". Als je foute data invoert, crasht hij of blijft hij stil.
  • Een setter ($card->setCard(...)) kan antwoord geven (return). Hij kan zeggen: "Error: Foute kleur".

Dit geeft ons meer controle en betere feedback tijdens het bouwen van de applicatie.

De Finale Code

Hieronder zie je hoe een modern index.php bestand eruitziet na deze module. Het is schoon, kort en krachtig.

<?php
// 1. We roepen de manager (Composer) aan.
// Dit is de enige 'require' die je ooit nog nodig hebt.
require_once 'vendor/autoload.php';

// 2. We geven aan welke classes we willen gebruiken.
// Dankzij Namespaces weet PHP precies welke bestanden hij moet pakken.
use Blackjack\Card;
use Blackjack\Player;

// 3. We starten het spel
echo "
Welkom bij Blackjack!
";

// Nieuwe speler maken
$player = new Player();
// We gebruiken de Bulk Setter voor feedback
echo $player->setPlayer("Sjaak") . "
";

// Nieuwe kaart maken
$card = new Card();
echo $card->setCard("Hearts", "Ace", "ace.png");

// 4. Resultaat tonen
echo "
";
echo "Speler: " . $player->getName();
echo " heeft kaart: " . $card->getCardName();
?>

Wat nu?

Je project staat nu als een huis. De fundering is professioneel. In de volgende modules gaan we dit huis verder inrichten. We gaan leren over Inheritance (overerving), zodat we niet steeds dezelfde code hoeven te typen voor dingen die op elkaar lijken.
