Introductie Inheritance

Welkom in Module 5! Tot nu toe heb je classes gemaakt die op zichzelf stonden. Maar naarmate je Blackjack-spel groeit, loop je tegen een probleem aan.

Je hebt een speler (jijzelf) en een dealer (de bank). Ze lijken heel erg op elkaar:

  • Ze hebben allebei kaarten in hun hand.
  • Ze hebben allebei een puntentotaal.
  • Ze kunnen allebei een kaart trekken.

Maar... ze zijn niet hetzelfde. De dealer heeft geen geld (fiches) en speelt volgens vaste regels, terwijl jij zelf mag kiezen.

Het Probleem: Hoe bouw je dit?

Als beginner heb je vaak de neiging om één super-class te maken die alles kan. Je stopt zowel de Mens als de Dealer in één bestand dat Player.php heet.

De Vergelijking: Het Zwitsers Zakmes

Stel je voor dat je één stuk gereedschap maakt voor alles. Een hamer, die ook een zaag is, en waar ook een schroevendraaier aan vast zit.

  • Als je wilt timmeren, zit de zaag in de weg.
  • Het ding wordt zwaar, onhandig en gaat snel stuk.

In programmeren noemen we zo'n class een God Class. Hij probeert God te spelen door alles tegelijk te zijn.

Voorbeeld: De "God Class" (Hoe het NIET moet)

Kijk eens naar de code hieronder. We proberen het gedrag van de Mens en de Dealer in één class te proppen met if-else statements.

class Player
{
    private string $role; // Is het 'human' of 'dealer'?

    // Deze property is logisch voor de Mens, maar onzin voor de Dealer.
    // De Dealer heeft namelijk geen eigen portemonnee.
    private int $balance = 1000;

    public function __construct(string $role)
    {
        $this->role = $role;
    }

    // Een gigantische functie met checks
    public function playTurn(): void
    {
        if ($this->role === 'human') {
            // Code voor de mens:
            // Wacht op knopdruk, mag splitsen, mag verdubbelen...
        }
        elseif ($this->role === 'dealer') {
            // Code voor de dealer:
            // Moet verplicht kaarten pakken tot 17 punten.
            // Mag NIET splitsen.
        }
    }

    public function placeBet(int $amount): void
    {
        // Weer een check... want een dealer kan niet wedden!
        if ($this->role === 'dealer') {
             // We moeten een error geven, want deze functie slaat nergens op voor een dealer
             echo "FOUT: De dealer zet geen geld in!";
             return;
        }
        $this->balance -= $amount;
    }
}

Waarom is dit zo slecht?

Misschien denk je: "Het werkt toch?". Maar kijk eens goed:

  1. Vervuiling: De Dealer sleept een variabele $balance mee die hij nooit gebruikt. Dat is zonde van het geheugen en verwarrend.
  2. Spaghetti code: Je hele code staat vol met if ($role === 'dealer'). Als je later een derde rol toevoegt (bijvoorbeeld een toeschouwer), moet je overal je code aanpassen.
  3. Kwetsbaarheid: Als je iets aanpast aan de logica voor de Dealer, maak je per ongeluk misschien de code voor de Mens kapot.

De Oplossing: Inheritance (Overerving)

In plaats van één rommelige class, gaan we werken met een basis en variaties. Dit heet Inheritance.

We maken één "Moeder Class" (de Basis) waar alles in zit wat ze gemeen hebben (kaarten hebben, punten tellen). Daarna maken we twee "Kinder Classes" die de unieke dingen doen.

Hoe ziet dat eruit?

We gaan dit in de volgende les bouwen, maar hier is het idee:

// 1. De Basis (Ouder)
// Hierin stoppen we ALLES wat gedeeld wordt (kaarten, hand, punten tellen)
class BasePlayer {
    protected array $hand = [];
}

// 2. De Mens (Kind)
// Deze "erft" alles van BasePlayer, maar voegt geld ($balance) toe.
class Human extends BasePlayer {
    private int $balance;
}

// 3. De Dealer (Kind)
// Deze "erft" ook alles van BasePlayer, maar heeft zijn eigen spelregels.
class Dealer extends BasePlayer {
    // Heeft GEEN $balance, want dat staat niet in de basis!
}

Waarom doen we dit?

  • Code Hergebruik (DRY): De logica voor "kaarten pakken" schrijf je maar één keer (in de basis). Zowel de Human als de Dealer kunnen dat meteen.
  • Schoon: De Dealer heeft geen $balance property meer. Hij is precies wat hij moet zijn, niets meer en niets minder.
  • Onderhoudbaar: Als je een foutje in de puntentelling oplost in de basis, is het meteen opgelost voor iedereen.

Samenvatting

  • Prop niet te veel functionaliteit in één class (vermijd de "God Class").
  • Als classes op elkaar lijken, maar net iets anders zijn, gebruik je Inheritance.
  • Je maakt een algemene basis class en specifieke sub-classes.
