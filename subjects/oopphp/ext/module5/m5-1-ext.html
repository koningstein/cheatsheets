Introductie Inheritance

Welkom in Module 5! Tot nu toe heb je classes gemaakt die helemaal op zichzelf stonden. Maar naarmate je Blackjack-spel groeit, zul je merken dat je code begint te rommelen. Je loopt tegen een ontwerpprobleem aan.

Denk eens aan de Speler (jijzelf) en de Dealer (de bank). Ze lijken ontzettend veel op elkaar:

  • Ze hebben allebei kaarten in hun hand.
  • Ze hebben allebei een puntentotaal.
  • Ze kunnen allebei een kaart trekken.

Maar... ze zijn niet 100% hetzelfde. De dealer heeft bijvoorbeeld geen eigen geld (fiches) en hij mag niet kiezen of hij past of koopt; hij moet verplicht de regels volgen.

Het Probleem: Hoe bouw je dit?

Als beginner heb je vaak de neiging om één super-class te maken die alles kan. Je stopt zowel de Mens als de Dealer in één bestand dat Player.php heet.

De Vergelijking: Het Zwitsers Zakmes

Stel je voor dat je één stuk gereedschap maakt voor alles. Een hamer, die ook een zaag is, en waar ook een schroevendraaier aan vast zit.

  • Als je wilt timmeren, zit de zaag in de weg (gevaarlijk!).
  • Het ding wordt zwaar, onhandig en gaat snel stuk.

In programmeren noemen we zo'n class een God Class. Hij probeert "God" te spelen door alles tegelijk te kunnen zijn. Dit willen we voorkomen.

Voorbeeld: De "God Class" (Hoe het NIET moet)

Kijk eens naar de code hieronder. We proberen het gedrag van de Mens en de Dealer in één class te proppen. We gebruiken een variabele $role om te onthouden wie het is.

De Class (Player.php)

Let op hoe we steeds moeten checken: "Ben ik een dealer? Doe dan dit. Ben ik een mens? Doe dan dat."

class Player
{
    private string $role; // Is het 'human' of 'dealer'?

    // Deze property is logisch voor de Mens, maar onzin voor de Dealer.
    // De Dealer heeft namelijk geen portemonnee, maar sleept deze variabele wel mee.
    private int $balance = 1000;

    public function __construct(string $role)
    {
        $this->role = $role;
    }

    // Een functie om geld in te zetten
    public function placeBet(int $amount): string
    {
        // Weer een check... want een dealer kan niet wedden!
        if ($this->role === 'dealer') {
             // We geven een foutmelding terug (geen echo!)
             return "FOUT: De dealer mag geen geld inzetten!";
        }

        $this->balance -= $amount;
        return "Inzet van $amount geplaatst. Nieuw saldo: $this->balance";
    }
}

Het Gebruik (index.php)

Als we deze class nu gebruiken, voelt het onlogisch aan.

// We maken een Dealer aan
$dealer = new Player('dealer');

// We proberen de dealer te laten wedden (wat eigenlijk niet kan)
$resultaat = $dealer->placeBet(50);

// We tonen het resultaat
echo $resultaat; // Toont: "FOUT: De dealer mag geen geld inzetten!"

Waarom is dit zo slecht?

Misschien denk je: "Het werkt toch?". Maar kijk eens goed naar de nadelen:

  1. Vervuiling: De Dealer heeft een variabele $balance die hij nooit gebruikt. Dat is zonde van het geheugen.
  2. Spaghetti code: Je hele code staat vol met if ($role === 'dealer'). Als je later een derde rol toevoegt (bijvoorbeeld een toeschouwer), moet je overal je code aanpassen.
  3. Kwetsbaarheid: Als je iets aanpast aan de logica voor de Dealer, maak je per ongeluk misschien de code voor de Mens kapot.

De Oplossing: Inheritance (Overerving)

In plaats van één rommelige class, gaan we werken met een basis en variaties. Dit heet Inheritance.

Denk aan biologie: Jij hebt eigenschappen van je ouders geërfd (oogkleur, haarkleur), maar je hebt ook je eigen unieke eigenschappen.

Hoe gaan we dit bouwen?

We maken één "Moeder Class" (de Basis) waar alles in zit wat ze gemeen hebben (kaarten hebben, punten tellen). Daarna maken we twee "Kinder Classes" die de unieke dingen doen.

De Classes

Hier zie je hoe schoon de code wordt. Let op het keyword extends.

// 1. De Basis (Ouder)
// Hierin stoppen we ALLES wat gedeeld wordt.
class BasePlayer {
    protected array $hand = [];

    public function getHand(): array {
        return $this->hand;
    }
}

// 2. De Mens (Kind)
// Deze "erft" alles van BasePlayer, maar voegt geld toe.
class Human extends BasePlayer {
    private int $balance = 1000;

    public function placeBet(int $amount): string {
        $this->balance -= $amount;
        return "Inzet geplaatst!";
    }
}

// 3. De Dealer (Kind)
// Deze "erft" ook alles van BasePlayer.
// Maar hij heeft GEEN placeBet() functie. Dat probleem is dus opgelost!
class Dealer extends BasePlayer {
    // Specifieke dealer regels komen hier...
}

Het Gebruik (index.php)

Nu kunnen we specifieke objecten aanmaken. De code beschermt ons nu tegen fouten.

$ik = new Human();
$bank = new Dealer();

// Dit werkt, want Human heeft deze functie
echo $ik->placeBet(50);

// Dit zou nu een echte PHP Foutmelding geven (Fatal Error),
// omdat de methode niet bestaat op de Dealer.
// $bank->placeBet(50); <--- Dit kan niet meer per ongeluk!

Waarom doen we dit?

  • Code Hergebruik (DRY): De logica voor "kaarten pakken" schrijf je maar één keer (in de basis). Zowel de Human als de Dealer kunnen dat meteen.
  • Schoon: De Dealer heeft geen $balance property meer. Hij is precies wat hij moet zijn.
  • Onderhoudbaar: Als je een foutje in de basis oplost, is het meteen opgelost voor iedereen.
