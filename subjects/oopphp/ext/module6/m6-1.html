Abstracte Classes

Het Probleem: Onvolledige Objecten

Tot nu toe hebben we een basis class Player en specifieke classes zoals Dealer en HumanPlayer. Maar stel jezelf de vraag: slaat het ergens op om "gewoon" een Speler aan te maken?

Een generieke Player weet niet of hij moet wachten op input of zelf moet spelen. Het is een onvolledig concept. Toch staat PHP toe dat we dit doen:

$player = new Player(); // Dit mag, maar zou niet moeten kunnen!

De Oplossing: Abstract Class

We kunnen voorkomen dat iemand direct een object maakt van een class door het keyword abstract toe te voegen. Een abstracte class dient alleen als blauwdruk voor andere classes.

abstract class Player
{
    protected array $hand = [];

    public function addCard(Card $card) {
        $this->hand[] = $card;
    }
}

Wat gebeurt er nu?

Als je nu probeert om een instantie te maken van de abstracte class, grijpt PHP in:

// DIT GEEFT NU EEN FOUTMELDING:
// Fatal error: Uncaught Error: Cannot instantiate abstract class Player
$player = new Player();

// DIT MAG NOG STEEDS:
// HumanPlayer erft van Player en is NIET abstract (concreet)
$human = new HumanPlayer("John");

Wanneer gebruik je Abstract?

Gebruik een abstracte class wanneer de ouder-class alleen bedoeld is om gedeelde code (zoals $hand en addCard) te bevatten, maar zelf nooit als object mag bestaan.

Type Hinting met Abstract Classes

Het mooie is dat je de abstracte class nog steeds kunt gebruiken als Type Hint. Je zegt hiermee: "Ik accepteer alles wat van Player afstamt".

Samenvatting

  • Abstract Class: Een class waarvan je geen objecten kunt maken (new Player() mag niet).
  • Doel: Dient als basis voor inheritance en dwingt structuur af.
  • Concrete Class: Een class die niet abstract is (zoals Dealer) en waarvan je wel objecten mag maken.
