Debugging & Laravel Debugbar

Hoe goed je code ook is, er gaat altijd wel iets mis. Laravel biedt standaard krachtige helpers (`dd`, `dump`), maar voor diepere inzichten in database queries, geheugengebruik en mails is de Laravel Debugbar onmisbaar.

Leeruitkomsten

  • Ik ken de standaard debugging functies (`dd`, `dump`, `Log`).
  • Ik kan de Laravel Debugbar installeren en gebruiken.
  • Ik kan de Debugbar gebruiken om performance problemen (N+1 probleem) op te sporen.

Standaard Debugging

In PHP ken je misschien var_dump(). Laravel maakt dit mooier:

  • dump($var): Toont de variabele, script gaat door.
  • dd($var): "Dump and Die". Toont de variabele en stopt het script direct.

Laravel Debugbar

Dit is een package van Barry vd. Heuvel die een balk onderin je browser toevoegt. Hier zie je alles wat er tijdens het request gebeurde.

Installatie (Dev only!)
composer require barryvdh/laravel-debugbar --dev

Wat kun je zien?

  • Queries: Zie exact welke SQL queries uitgevoerd worden en hoe lang ze duren. Dit is cruciaal om het N+1 probleem te vinden (als je 100 queries uitvoert voor 100 producten).
  • Route: Welke controller en middleware zijn gebruikt?
  • Views: Welke blade files zijn ingeladen?
  • Gate: Welke checks zijn gedaan (can/cannot)?
  • Mails: Zie een preview van verstuurde mails zonder ze echt te versturen.

Performance Optimalisatie

Als je pagina traag is, open je de 'Queries' tab. Zie je veel rode balken of honderden queries? Dan ben je waarschijnlijk vergeten om with() (Eager Loading) te gebruiken in je Eloquent query.


// Slecht (veroorzaakt N+1 queries)
$posts = Post::all();

// Goed (2 queries, ongeacht aantal posts)
$posts = Post::with('author', 'comments')->get();
