Git Workflow & Samenwerken

In een professionele omgeving werk je zelden alleen. Het direct committen naar de main branch is uit den boze. Je werkt met branches en Pull Requests (PRs). Dit zorgt ervoor dat code gecontroleerd kan worden (Code Review) en automatisch getest kan worden voordat het definitief wordt.

Leeruitkomsten

  • Ik begrijp de Feature Branch Workflow.
  • Ik kan een feature branch maken en switchen.
  • Ik weet wat een Pull Request (PR) is en waarom Code Reviews belangrijk zijn.

De Feature Branch Workflow

De gouden regel: De main branch is altijd deployable (stabiel).

  1. Branch maken: Voor elke taak (ticket, bugfix) maak je een nieuwe branch vanaf main.
  2. Werken: Je schrijft code en commit op je eigen branch.
  3. Push: Je uploadt je branch naar GitHub/GitLab.
  4. Pull Request: Je vraagt om je branch samen te voegen met main.
  5. Review & Merge: Een collega kijkt je code na. Na goedkeuring wordt het gemerged.

Basis Commando's

Nieuwe feature starten

# Update je main eerst
git checkout main
git pull

# Maak een nieuwe branch
git checkout -b feature/nieuwe-betaalmethode
Werk opslaan

git add .
git commit -m "Implementatie Mollie betalingen"
git push origin feature/nieuwe-betaalmethode

Code Review

Wanneer je een Pull Request opent, kijken collega's naar:

  • Functionaliteit: Doet het wat het moet doen?
  • Kwaliteit: Is de code leesbaar? Volgt het de standaarden?
  • Tests: Zijn er tests geschreven voor de nieuwe functie?
