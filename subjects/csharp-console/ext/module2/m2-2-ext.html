Namespaces & Using

Als je project groter wordt, krijg je steeds meer bestanden. Je wilt niet dat alles op één grote hoop ligt. Je wilt structuur. In C# gebruiken we hiervoor Namespaces.

Het Probleem: Dubbele Namen

Stel je voor dat jij een game maakt en je noemt een vijand Monster. Later download je code van een ander, en die heeft óók een class Monster.

Als je nu new Monster() typt, raakt C# in de war. Welke bedoel je? Die van jou, of die van de ander?

De Vergelijking: Achternamen

Stel dat er twee jongens in de klas zitten die allebei "Jan" heten. Als de leraar roept: "Jan, kom eens hier!", weten ze niet wie hij bedoelt.

Daarom gebruiken we achternamen: "Jan Jansen" en "Jan de Vries".

Een Namespace is eigenlijk de achternaam van je Class.

1. Een Namespace toevoegen

Laten we onze code een 'achternaam' geven. Dit doen we in het bestand van de class zelf.

Sinds C# 10 kunnen we dit heel kort opschrijven (File Scoped Namespace). Je zet het bovenaan met een puntkomma.

// BESTAND: DigitalPet.cs

// 1. We geven dit bestand een namespace (de 'achternaam')
namespace Tamagotchi;

public class DigitalPet
{
    public string Name { get; set; }
    // ... rest van de class ...
}

De volledige naam van je class is nu veranderd van DigitalPet naar Tamagotchi.DigitalPet.

2. Het gevolg: Program.cs is stuk!

Ga nu terug naar Program.cs. Je zult zien dat er opeens rode strepen staan onder DigitalPet.

// BESTAND: Program.cs

// FOUT: C# zegt: "Ik weet niet wat een 'DigitalPet' is."
// Hij zoekt naar de korte naam, maar het bestand heet nu 'Tamagotchi.DigitalPet'.
DigitalPet dino = new DigitalPet("Rex", isSleeping: true);

3. De Oplossing: Using

We kunnen dit op twee manieren oplossen.

Manier A: De volledige naam typen (Onhandig)

Tamagotchi.DigitalPet dino = new Tamagotchi.DigitalPet("Rex", isSleeping: true);

Dit werkt, maar is vreselijk veel typwerk.

Manier B: Using (De beste manier)

Met het keyword using zeg je tegen C#: "Als ik een naam typ die je niet kent (zoals DigitalPet), zoek dan even in deze doos."

// BESTAND: Program.cs

// 1. We importeren de namespace bovenaan
using Tamagotchi;

// 2. Nu snapt C# weer wat we bedoelen!
DigitalPet dino = new DigitalPet("Rex", isSleeping: true);

Console.WriteLine(dino.GetDescription());

Waarom werkt 'Console' dan wel?

Misschien is het je opgevallen dat Console.WriteLine het altijd doet, zonder dat je zelf using System; typt.

Dat komt omdat moderne C# projecten (sinds .NET 6) gebruik maken van Implicit Usings. Visual Studio voegt op de achtergrond stiekem al een lijstje met veelgebruikte namespaces (zoals System) toe. Jouw eigen namespace (Tamagotchi) staat daar niet bij, dus die moet je wél zelf toevoegen.

Wat kan er fout gaan? (Veelgemaakte Fouten)

Verkeerde plek

De using moet altijd helemaal bovenaan in je bestand staan. Nog voor je code begint.

Rode strepen oplossen

Zie je een rode streep onder een class naam?
1. Klik op de rode tekst.
2. Druk op Ctrl + . (punt).
3. Kies de optie using Tamagotchi;.
Visual Studio voegt de regel dan zelf bovenaan toe.

Samenvatting

  • Een Namespace geeft je code een uniek adres (zoals een achternaam).
  • Zodra je een class in een namespace zet, kan Program.cs hem niet meer zomaar vinden.
  • Gebruik using bovenaan je bestand om de namespace te importeren.
