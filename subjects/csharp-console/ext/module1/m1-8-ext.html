Named Arguments: Duidelijke Code

In de vorige lessen heb je geleerd hoe je methodes en constructors maakt met parameters. Zolang je maar één of twee parameters hebt (bijvoorbeeld alleen een naam), is dat prima te lezen.

Maar wat als je object ingewikkelder wordt? Wat als je 5 getallen moet meegeven bij het aanmaken? Dan wordt je code onleesbaar.

C# heeft hier een geweldige oplossing voor: Named Arguments.

Het Probleem: De Getallenbrij

Stel je voor dat we onze DigitalPet uitbreiden. Bij de geboorte (constructor) willen we niet alleen de naam weten, maar ook de energie, de honger en of hij slaapt.

class DigitalPet
{
    // ... properties ...

    // Een constructor met veel parameters
    public DigitalPet(string name, int energy, int hunger, bool isSleeping)
    {
        // ... code om alles in te stellen ...
    }
}

Als je dit object nu aanmaakt in Program.cs, ziet dat er zo uit:

DigitalPet dino = new DigitalPet("Dino", 50, 10, false);

Kijk naar die getallen: 50, 10, false. Weet jij over een maand nog wat die 10 betekent? Is het 10 honger? Of 10 levenspunten? En betekent false dat hij niet slaapt, of dat hij niet dood is?

Dit is onduidelijke code. Je moet steeds terugkijken naar de class om te weten wat de volgorde is.

De Oplossing: Named Arguments

Met Named Arguments mag je de naam van de parameter erbij typen als je de functie aanroept. Je zet de naam, gevolgd door een dubbele punt : en dan de waarde.

De Vergelijking: Het Formulier

Positional Arguments (De oude manier): Iemand geeft je een blaadje en zegt: "Schrijf hier je naam, dan je leeftijd, dan je adres." Als je de volgorde vergeet, gaat het mis.

Named Arguments (De nieuwe manier): Je krijgt een formulier met labeltjes: "Naam: ...", "Leeftijd: ...". Je weet precies wat je waar invult, en de volgorde maakt niet eens meer uit.

Hoe werkt het?

Je verandert niets aan de Class of de Constructor zelf. Je past het alleen aan op het moment dat je het gebruikt (in Program.cs).

// Duidelijk en leesbaar:
DigitalPet dino = new DigitalPet(
    name: "Dino",
    energy: 50,
    hunger: 10,
    isSleeping: false
);

Nu kan iedereen (ook je collega's) direct zien wat elk getal betekent. De code legt zichzelf uit ("Self-Documenting Code").

Voordeel: De volgorde maakt niet uit

Omdat je de namen gebruikt, hoef je je niet meer aan de volgorde te houden. C# zoekt zelf uit welk getal bij welke parameter hoort.

// Dit werkt ook prima, ook al staat het door elkaar:
DigitalPet fluffy = new DigitalPet(
    hunger: 0,
    isSleeping: true,
    name: "Fluffy",
    energy: 100
);

Wanneer gebruik je dit?

Je hoeft dit niet altijd te doen. Bij simpele functies is het overbodig.

Situatie Advies Voorbeeld
Weinig parameters Niet nodig Math.Max(5, 10);
Duidelijke context Niet nodig pet.Eat("Apple"); (Het is logisch dat Apple het voedsel is)
Veel getallen/bools Gebruiken! CreateUser(isAdmin: true, sendEmail: false);

⚠️ Typefouten

De naam die je typt voor de dubbele punt (bijvoorbeeld energy:) moet exact overeenkomen met de naam van de parameter in de constructor. Als je in de class int levens hebt staan, werkt energy: 50 niet.

Samenvatting

  • Als een methode of constructor veel onduidelijke parameters heeft (zoals getallen of true/false), wordt je code moeilijk leesbaar.
  • Gebruik Named Arguments (naam: waarde) om duidelijk te maken wat je invult.
  • Dit verhoogt de leesbaarheid en je mag de volgorde van argumenten zelf bepalen.
