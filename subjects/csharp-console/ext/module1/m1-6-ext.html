Getters: Berekende Eigenschappen

Tot nu toe gebruikten we properties om data op te slaan, zoals Energy of Name. Maar soms wil je informatie hebben die je niet direct opslaat, maar die je kunt berekenen aan de hand van andere gegevens.

Dit noemen we Computed Properties (Berekende Eigenschappen) of Read-Only Properties.

Het Probleem: Dubbele Data

Stel je voor dat we willen weten of ons huisdier moe is. Je zou een variabele bool IsTired kunnen maken.

  • Als je de energie verlaagt, moet je niet vergeten IsTired aan te passen.
  • Als je de energie verhoogt, moet je weer checken of hij nog moe is.

Als je dit één keer vergeet, heb je een bug: Je huisdier heeft 100 energie, maar IsTired staat nog op true. Dat klopt niet!

De Oplossing: Live Berekenen

In plaats van de status "Moe" op te slaan, kunnen we het gewoon aan het object vragen op het moment dat we het nodig hebben. We kijken gewoon naar de huidige energie.

De Vergelijking: Je Leeftijd

Op je paspoort staat je Geboortedatum (Opgeslagen Data). Er staat niet je Leeftijd op.

Waarom niet? Omdat je leeftijd elke dag kan veranderen. Als iemand vraagt hoe oud je bent, reken je dat op dat moment uit: Vandaag min Geboortedatum.

Hoe maak je een Computed Property?

Een computed property heeft alleen een get. Hij heeft geen set, want je kunt de uitkomst niet "instellen". De uitkomst is het resultaat van een som of check.

class DigitalPet
{
    // Dit is opgeslagen data (Read & Write)
    public int Energy { get; set; }

    // Dit is berekende data (Read Only)
    // We voegen logica toe in het get-blok
    public string Status
    {
        get
        {
            if (this.Energy > 50)
            {
                return "Happy";
            }
            return "Tired";
        }
    }
}

Het gebruik in Program.cs

Voor de gebruiker van de class lijkt het net een gewone variabele. Je gebruikt gewoon de punt.

DigitalPet dino = new DigitalPet();
dino.Energy = 10;

// We lezen de status uit.
// Op dit moment rekent C# uit: is 10 groter dan 50? Nee. Return "Tired".
Console.WriteLine(dino.Status); // Output: Tired

dino.Energy = 90;
// Nu vragen we het opnieuw. C# rekent weer: is 90 groter dan 50? Ja.
Console.WriteLine(dino.Status); // Output: Happy

De 'Fat Arrow' Syntax (=>)

Programmeurs zijn lui. Als een getter maar uit één regel code bestaat (bijvoorbeeld een simpele check), vinden we get { return ... } te veel typowerk.

Hiervoor is de Expression Body syntax bedacht, ook wel de "Fat Arrow" (dikke pijl) genoemd.

class DigitalPet
{
    public int Energy { get; set; }

    // Oude, lange manier:
    /*
    public bool IsAlive
    {
        get { return this.Energy > 0; }
    }
    */

    // Nieuwe, korte manier:
    public bool IsAlive => this.Energy > 0;
}

Het pijltje => betekent eigenlijk: "Deze property is gelijk aan de uitkomst van...".

Wat kan er fout gaan? (Veelgemaakte Fouten)

⚠️ Proberen te schrijven (Setten)

Omdat deze properties alleen een get (of een pijl) hebben, zijn ze Alleen-Lezen (Read-Only).

// FOUT! Je kunt dit niet instellen. Het is een resultaat.
pet.IsAlive = true;
// Error: Property 'IsAlive' cannot be assigned to -- it is read only

⚠️ Pijltje vs Is-Gelijk-Aan

Verwar => (berekend telkens opnieuw) niet met = (startwaarde, één keer).

  • public int Score = 10; (Start op 10, kan veranderen).
  • public int Score => 10; (Is ALTIJD 10, kan nooit veranderen).

Samenvatting

  • Gebruik Computed Properties voor data die je kunt afleiden van andere data (zoals leeftijd, totaalbedrag, status).
  • Ze hebben geen setter, dus je kunt ze niet handmatig aanpassen.
  • Gebruik de Fat Arrow => om je code korter en leesbaarder te maken bij simpele checks.
