Setters & Validatie

Het gevaar van Auto-Properties

Tot nu toe gebruikten we { get; set; }. Dat is makkelijk, maar niet altijd veilig. Stel dat iemand per ongeluk de energie van je huisdier op -50 zet, of op 9999? Een auto-property accepteert alles.

Om dit te voorkomen, moeten we Logica toevoegen aan de setter. Hiervoor stappen we over naar een Full Property.

1. Backing Fields

Bij een Full Property slaan we de data op in een private field (een variabele die alleen binnen de class bestaat). De publieke Property praat vervolgens met dit geheime veld.

class DigitalPet
{
    // Private Backing Field (kleine letter, vaak met underscore)
    private int _energy;

    // Public Property (Hoofdletter)
    public int Energy
    {
        get { return _energy; } // Geeft de waarde terug
        set { _energy = value; } // Schrijft de waarde weg
    }
}

2. Validatie toevoegen

In het set blok kunnen we nu een if-statement toevoegen. Het keyword value bevat de data die iemand probeert in te vullen.

public int Energy
{
    get { return _energy; }
    set
    {
        // Validatie: Energie mag niet lager dan 0
        if (value < 0)
        {
            Console.WriteLine("Let op: Energie kan niet negatief zijn! We zetten het op 0.");
            _energy = 0;
        }
        else if (value > 100)
        {
            _energy = 100; // Maximaal 100
        }
        else
        {
            _energy = value; // Geldige waarde, dus opslaan
        }
    }
}

Encapsulation

Dit concept heet Encapsulatie. Je beschermt de data (het backing field) door er een gecontroleerde toegangspoort (de property) voor te zetten. De buitenwereld weet niet hoe de data wordt opgeslagen, alleen dat het werkt.
